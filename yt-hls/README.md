# yt-hls

Минимальный локальный HLS-шлюз для YouTube.

**Одна функция:** на вход — id видео (`gofCdGw1VGA`), на выход — локальная
`.m3u8`-ссылка, которую можно вставить в Safari (или любой HLS-плеер) и
смотреть стрим напрямую, **без скачивания файла целиком**.

```
id видео ──▶ [ доступ к SABR ] ──▶ [ SabrStream тянет HD ] ──▶ [ ffmpeg → HLS ] ──▶ .m3u8
```

## Почему это вообще нужно

YouTube для обычных VOD больше не отдаёт ни HLS, ни DASH-манифест, ни рабочий
прогрессивный mp4 — только **SABR** (POST-протокол с PO Token), который Safari
не умеет. Поэтому «голой ссылки» для Safari не существует. Этот сервис делает
переводчик: получает SABR-поток (как делает плеер youthub) и **на лету**
переупаковывает его в HLS, который Safari играет нативно.

## Что взято из youthub (механизм доступа)

Целиком перенесены три куска, отвечающие за успешный доступ к SABR-стриму:

- **`pr_fetch.py`** — достаёт `playerResponse.streamingData`, обходя бот-стену
  липкой ротацией TLS/IP/endpoint-отпечатков (curl_cffi: chrome/firefox/safari).
- **`po_token.mjs`** — минтит content-bound PO Token через bgutils (без браузера).
- n-decipher `serverAbrStreamingUrl` через youtubei.js.

Дальше `googlevideo/SabrStream` тянет дорожки, а `ffmpeg -c copy -f hls`
переупаковывает их в плейлист + сегменты.

> **Safari-нюанс:** нативный HLS требует **H.264 + AAC**. Лучшие дорожки
> YouTube — VP9 + Opus, их Safari не играет. Поэтому сервер явно выбирает
> H.264-itag + AAC-itag (140). Если видео отдаёт только VP9/Opus — эта дорожка
> транскодируется (медленнее, редкий случай).

## Установка

```sh
cd yt-hls

# 1. Node-зависимости (+ патч googlevideo для startAtMs идёт в postinstall)
npm install

# 2. Python-окружение для pr_fetch.py (нужен curl_cffi)
python3.14 -m venv .venv          # или любой python ≥3.11
.venv/bin/pip install -r requirements.txt
# pr_fetch.py зовётся как python3.11 — дадим симлинк (на macOS реальный — 3.14):
ln -sf python3 .venv/bin/python3.11
```

Нужен системный `ffmpeg` (`brew install ffmpeg`).

## Запуск

```sh
node server.mjs                          # 127.0.0.1:8730 (только этот Mac)
PORT=9000 HOST=0.0.0.0 node server.mjs   # отдать в локальную сеть
```

Получить метаданные + ссылки и открыть в Safari:

```sh
curl http://127.0.0.1:8730/play/gofCdGw1VGA
# hls:      http://127.0.0.1:8730/hls/gofCdGw1VGA/index.m3u8
# download: http://127.0.0.1:8730/dl/gofCdGw1VGA

# для скриптов — JSON:
curl -H 'Accept: application/json' http://127.0.0.1:8730/play/gofCdGw1VGA | jq -r .hls
open -a Safari "$(curl -sH 'Accept: application/json' http://127.0.0.1:8730/play/gofCdGw1VGA | jq -r .hls)"
```

`/play` — это «resolve»-эндпоинт: он стартует сессию и возвращает то, что клиент
сам вычислить не может — **заголовок**, **длительность** и **реально отдаваемые
кодеки** (транскод VP9→H.264 честно показан как `h264`; `x=1` → `video: null`),
плюс сразу **обе** ссылки: `hls` (стрим) и `download` (файл). С `Accept:
application/json` отдаётся объект `{id, title, durationMs, video, audio, hls,
download}`.

Первый запрос на видео занимает ~5–15 с (обход бот-стены + PO Token + первый
сегмент), дальше плейлист отдаётся мгновенно и растёт по мере подкачки.

## Обрезка диапазона (`?from=&to=`, секунды)

К любому URL можно добавить `?from=&to=` — тогда отдаётся **конечный** VOD-клип
только этого диапазона (не всё видео). Диапазон тянется точечно (SABR
`startAtMs`), так что качается только он, а не весь ролик.

```sh
curl "http://127.0.0.1:8730/hls/gofCdGw1VGA/index.m3u8?from=35&to=144"
open -a Safari "http://127.0.0.1:8730/hls/gofCdGw1VGA/index.m3u8?from=4.0&to=12"
```

Точность **пер-край** и задаётся наличием точки в числе:

| значение | режим | что делает |
|----------|-------|------------|
| целое (`35`)   | приблизительно | рез по ближайшему ключевому кадру ≤ значения, **чистый `-c copy`** (без ре-энкода) |
| дробное (`35.7`) | точно | smart-cut: ре-энкодится **только** пограничный GOP этого края, всё остальное — copy |

Края независимы: `from=4.0&to=12` — начало точное, конец по ключевому кадру.

**Гарантии по A/V:** звук не ре-энкодится никогда (AAC-фреймы независимы,
режутся copy по своей ~21 мс сетке; точные края округляются к этой сетке).
Части (head-реэнкод / body-copy / tail-реэнкод) склеиваются на уровне плейлиста
через `#EXT-X-DISCONTINUITY`, а не `concat -c copy` — потоки нигде не
пере-базируются по отдельности, поэтому **рассинхрона видео/звука нет** (тело
клипа — побитово синхронный copy исходника; на реэнкоженном крае возможно
неощутимое смещение ≤ одного аудио-фрейма, оно не накапливается).

> **Про предупреждения в консоли.** «Многословный» декодер (ffplay/ffmpeg) на
> каждом стыке `#EXT-X-DISCONTINUITY` печатает `mmco: unref short failure` или
> `Packet corrupt` — это декодер согласовывает список опорных кадров между двумя
> независимо закодированными частями, **а не дефект**. Число сообщений = число
> стыков: целые края `35&60` → 0, один точный край → 1, оба точных `35.5&60.5`
> → 2. Каждая часть начинается со своего ключевого кадра, картинка верна с
> первого кадра; Safari (нативный HLS) сбрасывает декодер на стыке и вообще
> ничего не печатает. Убрать полностью можно лишь сплошным ре-энкодом всего
> клипа — что убивает весь смысл «без ре-энкода».

## VP9 без ре-энкода (`&vp9=1`)

Если у видео нет H.264 (только VP9), сервер по умолчанию **транскодирует** его в
H.264 — иначе Safari не сыграет. Это дорого по CPU. Флаг `&vp9=1` говорит: «мой
плеер умеет VP9 — не транскодируй, копируй как есть».

```sh
open -a "Google Chrome" "http://127.0.0.1:8730/hls/<id>/index.m3u8?from=4.0&to=12&vp9=1"
```

- **Только не-Safari плеер.** VP9 не играет в Safari **ни в каком** контейнере.
  `vp9=1` — это Chrome / Firefox / hls.js / ffplay / Android-WebView. На iOS любой
  «браузер» использует движок Safari → VP9 там не будет.
- **Другой контейнер.** VP9 нельзя положить в `.ts` (mpegts) — сервер переходит
  на **fMP4** (`init.mp4` + `.m4s` + `#EXT-X-MAP`, HLS v7). Стыки частей у fMP4
  чище mpegts — предупреждений `mmco` там нет.
- **Точные края работают** и для VP9: пограничный GOP реэнкодится через
  `libvpx-vp9` (поток остаётся VP9), тело — чистый copy. Запрещён лишь транскод
  **всего** фрагмента, а не точная подрезка края.
- **Целые края грубее, чем на H.264.** У VP9 ключевые кадры реже (GOP до ~5 с),
  так что приблизительный рез может промахнуться на несколько секунд. Нужна
  точность — используй дробные секунды (`from=4.0`).
- Действует **только на VP9-only видео**: если H.264 есть, он и выбирается, а
  `vp9=1` ни на что не влияет (обычный mpegts-путь). Флаг входит в ключ сессии.

## Только звук (`&x=1`)

Флаг `&x=1` отдаёт **только аудио-дорожку** (без видео). SABR при этом тянет
только звук (видео вообще не качается), ffmpeg мультиплексит один поток → лёгкий
audio-only HLS. Работает и на полном видео, и с обрезкой `?from=&to=`.

```sh
curl "http://127.0.0.1:8730/hls/<id>/index.m3u8?from=35&to=60&x=1"
```

- Всегда **AAC** (copy, если источник AAC; редкий Opus-only транскодируется в
  AAC) — играет везде, включая Safari.
- Обрезка звука **всегда точная и без ре-энкода**: у аудио нет ключевых кадров,
  каждый фрейм самостоятелен, поэтому copy-рез точен до сетки фреймов (~21 мс). Тут
  нет разницы «целое/дробное» и нет `#EXT-X-DISCONTINUITY` — один сплошной кусок.
- С `x=1` флаг `vp9=1` игнорируется (видео нет). Флаг входит в ключ сессии.

## Скачивание файлом (`/dl/<id>`, `&dl=1`)

Чтобы **сохранить** ролик на устройство (а не стримить), есть отдельная ссылка
`/dl/<id>` с теми же параметрами `?from=&to=`, `&x=1`, `&vp9=1`. Готовый плейлист
ремуксится в **один файл** через `-c copy` (без ре-энкода) и отдаётся с заголовком
`Content-Disposition: attachment` (имя файла = заголовок видео + диапазон).

```sh
curl -OJ "http://127.0.0.1:8730/dl/<id>?from=35&to=60"        # → <title> 35-60.mp4
curl -OJ "http://127.0.0.1:8730/dl/<id>?from=35&to=60&x=1"    # → <title> 35-60.m4a (только звук)
curl -OJ "http://127.0.0.1:8730/dl/<id>"                      # → весь ролик .mp4
```

- Контейнер: `.mp4` для видео, `.m4a` для `x=1`. `-OJ` у curl подхватывает имя из
  заголовка.
- **Скачивание переиспользует сессию плейбэка** — те же сегменты, что уже отдаются
  в `/hls`. `dl` не входит в ключ сессии.
- Готовую `/dl`-ссылку заодно отдаёт `/play/<id>` (поле `download`).
- **Целый ролик** (`/dl` без `from/to`) ждёт, пока подкачается **всё** видео (у
  полного HLS плейлист растёт, пока не будет `#EXT-X-ENDLIST`) — для длинных видео
  это десятки секунд и сотни МБ. Обрезанный клип готов сразу.
- На точных краях видео (`from=35.7`) в скачанном mp4 возможен один косметический
  кадр-артефакт на стыке (тот же компромисс, что и `mmco`-предупреждения в
  стриминге: одна avcC-дорожка mp4 против сменившихся параметров кодека). При целых
  краях (чистый copy) файл идеально чистый.

## Эндпоинты

| метод | путь | назначение |
|-------|------|------------|
| `GET` | `/play/<id>[?from=&to=][&x=1][&vp9=1]` | стартует сессию, возвращает метаданные (title, длительность, честные кодеки) + ссылки `hls` и `download` (`Accept: application/json` → JSON) |
| `GET` | `/hls/<id>/index.m3u8[?from=&to=][&x=1][&vp9=1]` | HLS-плейлист (без `from/to` — растущий EVENT на всё видео; с ними — конечный VOD-клип) |
| `GET` | `/hls/<id>/<segment>` | медиа-сегмент (`segNNNNN.ts`, или при `vp9=1` — `init.mp4` + `segNNNNN.m4s`) |
| `GET` | `/dl/<id>[?from=&to=][&x=1][&vp9=1]` | скачать один файл (`.mp4`, или `.m4a` при `x=1`), `Content-Disposition: attachment` |
| `GET` | `/healthz` | `ok` |

## Конфигурация (env)

| переменная | по умолчанию | что делает |
|------------|--------------|------------|
| `PORT` | `8730` | порт |
| `HOST` | `127.0.0.1` | интерфейс (`0.0.0.0` — отдать в сеть) |
| `YT_HLS_PYTHON` | `./.venv/bin/python3.11` | интерпретатор для `pr_fetch.py` |
| `YT_HLS_SESSIONS` | `./sessions` | куда ffmpeg пишет плейлисты/сегменты |
| `HTTPS_PROXY` | — | проксируется в `pr_fetch.py` (ротация `-proxy` стратегий) |

## Ограничения

- Работает только пока сервис запущен и только локально (URL — `localhost`,
  если не выставить `HOST=0.0.0.0`). Это не публичная ссылка.
- Под капотом та же бот-стена: если IP временно зарейтлимичен — `pr_fetch`
  вернёт ошибку, надо подождать или сменить прокси.
- Сессии переиспользуются и сами убираются после 30 мин простоя.
